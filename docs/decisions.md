# Decisions

## 概要

本ファイルでは、本プロジェクトで行った主な設計判断と、その理由を整理しています。  
「何を使ったか」だけでなく、「なぜそうしたか」を明文化することを目的としています。

---

## 1. Jetpack Compose を採用した理由

### 判断
UI 実装には Jetpack Compose を採用しました。

### 理由
- モダン Android 開発の中心的な UI 技術であること
- 状態と UI の対応関係を明確にしやすいこと
- 画面構築や状態管理の見通しを良くしたかったこと
- 今後の UI テストや設計改善と相性が良いこと

### 期待した効果
- UI の責務整理
- 状態駆動の画面構築
- 画面ごとの保守性向上

---

## 2. Single Activity + Navigation Compose を採用した理由

### 判断
Activity を増やさず、Single Activity + Compose Navigation 構成を採用しました。

### 理由
- Compose と相性が良いこと
- 画面遷移管理を一元化しやすいこと
- Activity ごとの責務分散を避けたかったこと

### 期待した効果
- 画面遷移の整理
- ルーティングの見通し向上
- UI 構成の一貫性確保

---

## 3. Multi Module 構成を採用した理由

### 判断
アプリを `app / core / feature / sync` に分割しました。

### 理由
- 機能ごとの責務を整理したかったこと
- 共通処理と feature 固有処理を分離したかったこと
- コードベースの見通しを良くしたかったこと

### 期待した効果
- 依存関係の明確化
- 機能単位の保守性向上
- テスト対象の切り分け

---

## 4. Hilt を採用した理由

### 判断
依存注入には Hilt を採用しました。

### 理由
- Android で一般的な DI 構成であること
- ViewModel や Worker と接続しやすいこと
- Module 化した依存の管理を整理しやすいこと

### 期待した効果
- 依存関係の明示化
- テストや差し替えのしやすさ向上
- インスタンス生成責務の分離

---

## 5. Retrofit / OkHttp / Moshi を採用した理由

### 判断
API 通信には Retrofit / OkHttp / Moshi を採用しました。

### 理由
- Android の API 通信で広く使われる構成であること
- 通信処理、JSON 変換、ログの責務を分けやすいこと
- 学習コストと実装コストのバランスが良いこと

### 期待した効果
- ネットワーク層の整理
- データ変換の明確化
- Repository 実装の簡潔化

---

## 6. DataStore を採用した理由

### 判断
設定値や軽量な永続データには DataStore を採用しました。

### 理由
- セッションや画面状態の保存に適していること
- SharedPreferences より現代的で整理しやすいこと
- Flow ベースで扱いやすいこと

### 扱うもの
- ログイン状態
- セッション情報
- Availability の選択状態
- 軽量な画面状態

---

## 7. Room を導入した理由

### 判断
業務データには Room を導入しました。

### 理由
- スケジュールのような一覧 / 詳細 / 絞り込みを伴うデータは、DataStore より DB のほうが適していること
- ローカルキャッシュの責務を明確にしたかったこと
- API 直読み依存を減らして、データ取得の構造を整理したかったこと

### 導入前の課題
- 業務データと軽量設定が混ざりやすい
- 一覧や詳細の責務が曖昧
- ViewModel が API 取得と表示責務を持ちやすい

### 導入後の変化
- 一覧 / 詳細を Room ベースに整理
- observe / sync ベースの構成へ移行
- DataStore と Room の責務分離が明確になった

---

## 8. DataStore と Room を分けた理由

### 判断
ローカルデータの責務を分離しました。

### 方針
- DataStore = 設定 / 軽量状態
- Room = 業務データ

### 理由
両者は得意な用途が異なるためです。  
この責務を分けることで、ローカルデータ設計の意図を明確にできます。

### 効果
- 保存対象の判断がしやすい
- Repository の責務が整理しやすい
- 将来の拡張がしやすい

---

## 9. WorkManager を採用した理由

### 判断
定期同期や継続的なバックグラウンド処理には WorkManager を採用しました。

### 理由
- Android で継続処理を扱う標準的な選択肢であること
- スケジュール同期の題材と相性が良いこと
- foreground に依存しない処理を整理したかったこと

### 効果
- 同期処理を UI から分離できる
- バックグラウンド処理の責務が明確になる

---

## 10. テストを層ごとに分けた理由

### 判断
Unit Test / DAO Test / UI Test / Worker Test を分けて導入しました。

### 理由
- 責務の異なる対象を、同じ種類のテストで無理に確認しないため
- まずは速い unit test から厚くしやすいため
- DB や Worker のような対象は専用テストのほうが明確だから

### 効果
- テストの意図が分かりやすい
- 追加時の方針がぶれにくい
- 継続的な改善の足場になる

---

## 11. GitHub Actions を導入した理由

### 判断
GitHub Actions を用いて build / test の自動化を追加しました。

### 理由
- ローカル確認だけに依存しないようにするため
- build / test の実行漏れを減らすため
- 公開リポジトリとして、継続開発の形を見せたかったため

### 効果
- push / PR 時の確認自動化
- GitHub 上での実行結果確認
- 「GitHub を使っている」だけでなく、「GitHub Actions で自動化している」と言える状態

---

## 12. 正直に未完了部分を残している理由

### 判断
実装済みと未完了部分を README や docs で分けて示す方針にしています。

### 理由
- 実装していない技術まで採用済みと見せないため
- 現状の課題認識も含めて見せるほうが誠実だから
- 改善過程自体がプロジェクトの価値になるため

### 例
- セキュリティ強化は今後の課題
- テストは導入済みだが、まだ拡充余地あり
- 一部 feature は今後改善予定

---

## 13. サーバー側を最小実装にとどめた理由

### 判断
本プロジェクトでは、サーバーサイドは動作確認に必要な最小限の実装にとどめています。

### 理由
- 主題を Android アプリ側の設計・実装・改善に置いているため
- リソースを Android 側の責務整理、テスト、CI に集中させたかったため
- ポートフォリオとして Android 側の考え方を伝えやすくするため

### 効果
- Android アプリの設計改善に注力できた
- ローカルデータ設計や UI / テストの改善を明確に示しやすくなった

---

## 14. 改善過程を README / docs に残す理由

### 判断
完成機能だけでなく、改善の経緯や判断理由もドキュメントに残しています。

### 理由
- 最終形だけでは、何を課題と認識して改善したかが伝わりにくいため
- 技術選定の背景を説明できると、コードだけでは伝わらない意図が見えるため
- 学習や改善のプロセス自体も成果物だと考えているため

### 効果
- 読み手が設計意図を理解しやすい
- 実装の取捨選択を説明しやすい
- 継続的に改善しているプロジェクトとして見せやすい

---

## まとめ

本プロジェクトで特に重要な設計判断は次の3つです。

1. モジュール分割による責務整理
2. DataStore と Room の責務分離
3. テストと CI を含めた継続開発前提の構成

これらにより、単に動くアプリではなく、  
設計と改善の意図を持った Android プロジェクトにすることを目指しています。
